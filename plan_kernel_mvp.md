# Shiro Project Kernel MVP Plan

## 1. 项目定位

本项目不是普通聊天机器人，而是一个面向 Socionics 分析场景的结构化判断系统：

- 用户输入 3-5k 字文本
- 系统输出 1-2k 字个性化分析判断
- 支持多轮符合逻辑的追问和解释
- 输出需强符合设定问答框架
- 前后端共同控制输入、输出和付费流程

核心路线：

```text
先 Kernel 后包装
Kernel 稳定后，再接前端、小程序、支付和后台数据系统
```

## 2. 模块拆分

### Kernel1: 单次分析引擎

目标：

- 输入指定文本
- 输出对应的类型判定和分析
- 结果尽量结构化、可校验、可复盘

建议输出结构：

```json
{
  "type_judgement": "候选类型",
  "confidence": 0.72,
  "evidence": ["关键证据1", "关键证据2"],
  "excluded_types": ["排除类型1", "排除类型2"],
  "uncertainties": ["不确定点1"],
  "follow_up_questions": ["建议追问1"],
  "final_report": "面向用户的自然语言分析"
}
```

实现原则：

- 先输出 JSON，再生成自然语言报告
- 不让模型完全自由发挥
- 类型判断必须带证据、置信度和不确定点
- 避免医学化、诊断化、绝对化表述

### Kernel2: 多轮对话引擎

目标：

- 支持多轮符合逻辑的问答
- 保持前后结论一致
- 能根据新输入修正判断

不建议简单把全部聊天历史塞入上下文。建议维护结构化状态：

```json
{
  "case_id": "唯一会话ID",
  "user_profile_summary": "用户文本摘要",
  "confirmed_findings": ["已确认判断"],
  "candidate_types": ["候选类型"],
  "rejected_types": ["已排除类型"],
  "open_questions": ["待确认问题"],
  "risk_flags": ["内容风险标记"],
  "conversation_summary": "多轮对话摘要"
}
```

多轮流程：

```text
当前用户输入
  ↓
输入安全检查
  ↓
读取 case_state
  ↓
生成回答
  ↓
更新 case_state
  ↓
输出安全检查
  ↓
返回用户
```

## 3. 工程路线

### Phase 1: Kernel 原型

目标：

- 实现 Kernel1
- 输入 3-5k 字文本
- 输出 800-1500 字分析报告
- 支持结构化 JSON 结果
- 保存请求、响应和错误日志

验收指标：

- 单用户可稳定完成一次分析
- 输出字段完整
- 类型判断有证据链
- 失败时有明确错误信息

### Phase 2: Kernel2 多轮

目标：

- 增加 case_id
- 增加 case_state
- 支持 3-5 轮追问
- 能引用前文结论
- 能在新证据下修正判断

验收指标：

- 多轮回答不明显跑偏
- 不重复长篇输出相同内容
- 能区分已确认结论和待确认问题

### Phase 3: 小范围封装测试

目标：

- 用 GitHub 或简单 Web 页面封装测试入口
- 收集真实样本
- 人工复盘输出质量
- 建立基础评测集

建议先不要接小程序和支付。

需要验证：

- 用户是否愿意阅读 1-2k 字报告
- 类型判断是否稳定
- 多轮是否会跑偏
- 平均延迟是否可接受
- 输入/输出拦截是否有效

### Phase 4: 小程序 + SDK + 后台

目标：

- 接入小程序前端
- 接入登录、订单、支付 SDK
- 建立后台数据和复盘系统
- 增加用户数据删除、订单追踪、退款处理能力

风险：

- 内容合规
- 用户数据合规
- 支付 SDK 合规
- 小程序审核风险

## 4. 推理架构建议

建议采用 Pipeline，而不是一开始采用复杂 Agent：

```text
用户输入
  ↓
输入安全检查
  ↓
文本预处理/摘要
  ↓
Kernel1 类型分析 JSON
  ↓
结果校验
  ↓
自然语言报告生成
  ↓
输出安全检查
  ↓
返回用户
```

理由：

- Pipeline 更可控
- 更容易测试
- 更容易定位问题
- 更适合固定理论框架输出

Agent 框架可以后置，用于：

- 多步骤资料整理
- 自动追问
- 后台复盘辅助
- 长文本分段分析

但 MVP 阶段不要让 Agent 主导核心判断。

## 5. 模型方案

当前设想：

```text
本地模型: Qwen3.6-27B
用途: 核心分析与多轮问答
```

建议：

- 初期不要追求超长上下文
- 单轮上下文目标控制在 8k-16k token
- 输入 3-5k 中文字，加系统设定、状态和输出约束后基本够用
- 输出控制在 800-1500 字，后续再扩到 1-2k 字

硬件建议：

| 阶段 | GPU 建议 | 说明 |
| --- | --- | --- |
| 原型验证 | 24GB 显存 + 4bit 量化 | 可试，但速度和质量可能一般 |
| 推荐起步 | 48GB 显存 | 更适合 27B 模型和轻并发 |
| 稳定生产 | 80GB 显存 | 更适合长上下文和更高吞吐 |
| 后期扩容 | 多 GPU | 有收入和稳定流量后再考虑 |

## 6. 并发与排队

当前目标：

```text
支持同时对话 2 人
输入 3-5k 文本
输出 1-2k 个性化分析判断
```

建议不要一开始做真实高并发，而是：

- 增加请求队列
- 限制单用户并发
- 增加任务状态
- 增加超时机制
- 增加失败重试

基础队列状态：

```text
queued -> running -> completed
queued -> running -> failed
queued -> cancelled
```

用户侧体验：

- 显示排队中
- 显示分析中
- 支持超时提示
- 失败后允许重新提交

初期判断：

```text
用户少时，排队机制优先于横向扩容。
实际产生营收后，再提升 GPU、并发和吞吐。
```

## 7. 合规和风控

### 输入控制

需要拦截：

- 违法违规内容
- 明显隐私泄露
- 自伤或伤害他人内容
- 绕过系统设定的攻击性 prompt
- 要求输出歧视、诊断、操控他人的内容

### 输出控制

需要避免：

- 绝对化人格判断
- 医疗或心理疾病诊断
- 职业筛选式歧视判断
- 对用户作价值高低评价
- 引导用户做高风险决定

推荐表达方式：

```text
根据目前文本，更倾向于...
证据主要来自...
仍存在以下不确定点...
这不是医学或心理诊断，只能作为自我理解和交流风格分析参考。
```

### 后端总控

后端必须拥有最终控制权：

- 统一 prompt
- 统一 schema
- 统一安全策略
- 统一日志记录
- 统一输出审核

前端只做体验和基础限制，不能把安全控制完全放在前端。

## 8. 微调策略

微调不是第一步。

优先级应为：

1. 高质量 system prompt
2. Socionics 理论知识库/规则库
3. 固定输出 schema
4. 人工标注测试集
5. 结果一致性评测
6. 必要时再做 LoRA/QLoRA 微调

只有当 prompt、schema、样本和评测都成熟后，微调才有明确方向。

## 9. MVP 验收指标

第一版目标：

- 支持 1 个用户提交 3-5k 文本
- 60 秒内返回结构化分析
- 输出 800-1500 字
- 包含类型判断、证据、置信度、不确定点和建议追问
- 支持 3-5 轮追问
- 每次会话有 case_id
- 输入和输出都有拦截
- 后台保存日志，支持人工复盘

第二版目标：

- 支持排队机制
- 支持同时 2 人感知上并行使用
- 增加基础后台
- 增加样本评测集
- 增加失败重试和超时控制

第三版目标：

- 接小程序
- 接付费 SDK
- 接用户系统
- 接订单和数据后台
- 做正式合规检查

## 10. 当前优先任务

建议下一步按以下顺序执行：

1. 定义 Kernel1 输入输出 schema
2. 写第一版系统 prompt
3. 准备 20-50 条测试样本
4. 实现本地推理 API
5. 实现任务队列
6. 实现结果日志
7. 实现 Kernel2 case_state
8. 做小范围 Web 测试入口

核心原则：

```text
先证明 Kernel 能稳定、可控、可测地完成分析。
再考虑前端、小程序、支付和商业包装。
```
